→ ✅ 対応済み / @wait の setTimeout を _waitTimer で追跡・キャンセルするよう修正（旧タイマーが残存して新ラベルで誤発火していた）。pool_end・ch4out_end は @end 有り、ch4out_kotoha_piano・ch4out_mahiru_river・chapter7_sakura_break は選択肢ジャンプ先ラベルのため元々問題なし。

  Grokによる解説
    1. Label "xxx" ran out of commands without @end/@jump ・showing END意味:・
    ラベル（ch4out_end や pool_end など）の最後に、@end
    または @jump
    （または return 相当の命令）が書かれていないのに、命令が終わってしまった。
    エンジンが「ここで終わり」と判断して自動的にEND画面を表示しています。
    これは軽い警告ですが、意図せずゲームが終わってしまう原因になることがあります。

    2. _executeNext called at end of label "xxx" while choices are visible ・ignoring意味:・
    選択肢（@choice
    やメニュー）がまだ画面に表示されている状態で、ラベル（ch4out_kotoha_piano、ch4out_mahiru_river、chapter7_sakura_break など）の命令が終わってしまった。
    エンジンが「選択肢が出ているのに次に進もうとした」と判断して、無視しています。
    これが一番問題になりやすいです。選択肢を選ぶ前に勝手に次のラベルに進もうとして、ゲームが止まったり、選択できなくなったりする原因になります。

突然ENDになる件について質問した結果。
    TyranoScript（ティラノスクリプト）では、ラベルの最後に @end または @jump（または [jump]）が正しくないと、エンジンが「命令が尽きた」と判断して自動的にEND画面を表示します。特に「放置してから進める」と発生しやすい理由は：選択肢（[choice] や [glink] など）がまだ画面に残ったまま、ラベルが終了してしまう。
    放置すると、ブラウザのJavaScript実行が少し遅れたり、イベントのタイミングがずれたりして、エンジンの「次に進む処理」がおかしくなる。
    チャプター終了の処理（次の章へのジャンプ）が、選択肢の後処理が不十分な状態で呼ばれている可能性が高い。

    つまり、選択肢の後にジャンプ命令が漏れている or チャプター終了ラベルに @end
    / @jump
    が書かれていないのが主な原因です。
